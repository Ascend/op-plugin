# aclnn_extension Example Description

This example project demonstrates **custom operator integration through structured code generation**. It shows how to use YAML descriptions and a code generation tool to integrate custom ACLNN operators into the PyTorch NPU ecosystem and generate adaptation code that can be compiled into Ascend Extension for PyTorch (`torch_npu`).

## Application Scenarios

This example project applies to the following scenarios:

- **Integrating custom ACLNN operators**: CANN provides ACLNN APIs (such as `aclnnFastGelu`), and you want to expose them as PyTorch custom operators in `torch_npu`.
- **Structured operator adaptation**: 
  The ACLNN operator semantics are consistent with the target ATen IR. The adaptation layer **contains no complex logic except for output tensor allocation, where the shape and data type are determined through data type deduction from inputs**. 
  When these conditions are met, YAML can be used for structured description, and the tool can automatically generate C++ adaptation code, eliminating the need to manually implement the adaptation layer.

If the operator semantics differ from ATen, require complex shape/type inference, or need additional logic in the adaptation layer, you must use a non-structured approach (such as `opapi`).

## Environment and Dependencies

- **Runtime/Build dependencies**: PyTorch, `torch_npu`, and CANN.
  For installation and version requirements, see the [torch_npu Installation Guide](https://gitcode.com/Ascend/pytorch#Installation).

## Quick Execution of the Example Project

To quickly run this example project without modifying the operator, execute the following commands in sequence within the `aclnn_extension` directory:

```bash
# 1. Run gen.sh to generate adaptation code
bash gen.sh npu_custom.yaml

# 2. Build the wheel package by running setup.py and install the package
python setup.py bdist_wheel
cd dist
pip install aclnn_extension*.whl

# 3. Run the test script
cd ..
cd test && python test_npu_fast_gelu_custom.py
```

A successful test run indicates that the example project has executed correctly, and the `torch_npu.npu_fast_gelu_custom` operator can be used directly.

## Procedure

### 1. Writing the Custom Operator YAML (Structured Description)

In the `custom` section of `npu_fast_gelu.yaml`, define a strict one-to-one mapping between ATen IR and ACLNN. 
A YAML is considered structurally compatible if **the ACLNN API corresponding to `opapi` is semantically consistent with the ATen IR, and the adaptation layer contains no complex logic except for allocating the output tensor**.

The code sample (forward + backward) is as follows:

```yaml
custom:
  - func: npu_fast_gelu_custom(Tensor self) -> Tensor
    op_api: all_version
    gen_opapi:
      out:
        size: self
        dtype: self
      exec: aclnnFastGelu
  - func: npu_fast_gelu_custom_backward(Tensor grad, Tensor self) -> Tensor
    op_api: all_version
    gen_opapi:
      out:
        size: grad
        dtype: grad
      exec: aclnnFastGeluBackward
```

- `func`: The operator signature exposed on the PyTorch side (ATen IR format).
- `gen_opapi.out`: Input (such as `self` or `grad`) used to deduce the shape and data type of the output tensor. 
- `exec`: The ACLNN API to be called.

### 2. Execute Code Generation

Run the following command in the `aclnn_extension` directory:

```bash
bash gen.sh npu_custom.yaml
```

The script automatically generates adaptation code based on the PyTorch version in the current environment, requiring no additional user intervention.

### 3. Build the whl Package and Run Tests

After Step 2 is completed (that is, `gen.sh` has been executed), run `setup.py` to build the project. The `aclnn_extension*.whl` file will be generated under the `./dist/` directory:

```bash
python setup.py bdist_wheel
cd dist
pip install aclnn_extension*.whl
```

After installation, you can call the integrated custom operator (such as `torch_npu.npu_fast_gelu_custom`) in your code. For example, verify it using the test case provided in this project:

```bash
cd test
python test_npu_fast_gelu_custom.py
```

Passing the test indicates that the operator has been correctly integrated and the results are consistent with the reference implementation.

Note that the test script must not be executed within the `aclnn_extension` directory, as the `__init__.py` path may interfere with execution.

## Customization and Extension Guide

When adapting your own operator, **the core step is to replace the YAML content with your own operator definition**:

- **Operator YAML** (such as `npu_custom.yaml` or a custom `my_op.yaml`): Define your own `func`, `gen_opapi.out`, and `exec` (ACLNN API name) in the `custom` section following the formatting style used in this project.
- **gen.sh parameters**: If you use a new file name (such as `my_op.yaml`), run `bash gen.sh my_op.yaml`.
- **Test script**: Modify or add tests under the `test/` directory and call your exposed operator for numerical validation.

The workflow remains unchanged: run `gen.sh` first (with your YAML file), then run `setup.py` to build the `aclnn_extension*.whl`. After installation through `pip`, the operator can be used directly.

## gen.sh Structure and Code Generation Instructions

This section describes the structure of the `gen.sh` script in the current directory and the code generation commands it uses. It is intended for users who require customization, such as modifying output paths, adding or removing steps, or switching YAML files.

### Script Parameters and Environment

- **Input parameters**: `gen.sh` accepts two parameters (the second is optional).
  - `$1`: The operator YAML file (required), such as `npu_custom.yaml`.
- **Working directory**: The script changes the directory to its own location (`aclnn_extension/`), and all subsequent paths are relative to this directory.
- **Version and directory name**: The script parses `PYTORCH_VERSION` (such as `2.7.0`) from `torch.__version__`, and derives `PYTORCH_VERSION_DIR` (such as `v2r7`) for paths such as `op_plugin/config/v2r7/`.
- **Environment variables**: The script sets and exports the following variables:
  - `PYTORCH_VERSION`: PyTorch version number.
  - `PYTORCH_CUSTOM_DERIVATIVES_PATH`: Path to the generated derivatives file used by `autograd` and related components.
  - `ACLNN_EXTENSION_PATH`: Root directory of the current example project.
  - `ACLNN_EXTENSION_SWITCH="TRUE"`: Flag indicating ACLNN extension mode (used by certain `torchnpugen` modules for branching).
- **Created directories**: If they do not exist, the script creates `build`, `op_plugin`, and `torch_npu`.

### Code Generation Commands (Execution Order)

The script calls the following `torchnpugen` modules in sequence, each corresponding to a code generation stage. You can add, delete, or modify parameters as needed.

| Order| Module| Purpose| Key Parameters|
|------|------|------|----------|
| 1 | `torchnpugen.gen_op_plugin_functions` | Generates and processes `op_plugin_functions.yaml` (ATen IR and version metadata)| `--version`, `--output_dir` (such as `op_plugin/config/v2r7/`), `--source_yaml` (input operator YAML)|
| 2 | `torchnpugen.gen_derivatives` | Executes only when `$2` is provided; generates the forward/backward binding file `derivatives.yaml`| `--version`, `--output_dir`, `--source_yaml` (derivatives YAML)|
| 3 | `torchnpugen.gen_op_backend` | Generates `op_plugin` APIs and routing layers (such as `OpInterface` and `OpApiInterface`)| `--version`, `--output_dir` (`op_plugin/`), `--source_yaml` (generated `op_plugin_functions.yaml`), `--deprecate_yaml`|
| 4 | `torchnpugen.struct.gen_struct_opapi` | Generates structured ACLNN adaptation code (such as `StructKernelNpuOpApi.cpp`)| `--output_dir` (`op_plugin/ops/opapi/`), `--native_yaml`, `--struct_yaml` (operator YAML)|
| 5 | `torchnpugen.gen_backend_stubs` | Generates `torch_npu` backend stub code under `csrc/aten` based on `test_native_functions.yaml`.| `--output_dir`, `--source_yaml` (`test_native_functions.yaml`), `--impl_path`, `--op_plugin_impl_path`, `--op_plugin_yaml_path`|
| 6 | `torchnpugen.autograd.gen_autograd` | Generates `autograd`-related code under `csrc/aten` and `autograd` based on `test_native_functions.yaml`.| `--out_dir`, `--autograd_dir`, `--npu_native_function_dir` (`test_native_functions.yaml`)|

- **Steps 1 through 4** use the operator YAML and the generated `op_plugin_functions.yaml` to produce output under `op_plugin/`, which represents the core of ACLNN adaptation.
- **Steps 5 and 6** use `test_native_functions.yaml` to generate `torch_npu`-side registration and `autograd` integration code.

### Customization Options

- **Input YAML**: Modify script arguments or internal variables (`$YAML_FILE`, `$DERIVATIVES_YAML_FILE`), or introduce new variables and pass them to `--source_yaml` / `--struct_yaml`.
- **Output paths**: Modify `OUTPUT_DIR`, `OPAPI_OUTPUT_DIR`, `--output_dir`, `--out_dir`, or `--autograd_dir` to redirect generated artifacts (must remain consistent with `setup.py` build paths).
- **PyTorch version**: Automatically inferred from the environment. For fixed versions, modify `PYTORCH_VERSION` and `PYTORCH_VERSION_DIR` directly.

These modifications allow flexible extension or trimming of the `gen.sh` pipeline while preserving the overall workflow.

## Summary

- **Application scope**: Structured ACLNN operator integration where semantics align with ATen and the adaptation layer only allocates the output tensor. 
- **Workflow**: Run `gen.sh npu_custom.yaml` to generate adaptation code, then run `setup.py` to build `aclnn_extension*.whl`, and install the package using `pip install` to use the operator. 
- **Customization**: Replace the content in the YAML file with your own operator definition, pass your YAML filename to `gen.sh`, and follow the same sequence of running `gen.sh` before running `setup.py`.
